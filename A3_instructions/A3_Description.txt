Object-Oriented Python Design
Your Python code for A3 must demonstrate object-oriented design and clean modular structure consistent
with the Model–View–Controller (MVC) pattern introduced in class.
Ensure that the Python side of your project is a clean Model–View–Controller structure:
• MVC Model: Player, and any other classes (e.g. Treasure) you choose to implement (Direct wrappers around C functions using ctypes.)
• MVC Controller: GameEngine (Coordinates model and view, manages ﬂow of the game.)
• MVC View: GameUI (Handles all rendering, display updates, and user input.)
• See class discussions in the lectures for more details on the MVC pattern
Design Expectations
1. Encapsulation:

2

Each class should expose clear, minimal public methods. Avoid leaking implementation details (e.g.,
C pointers or ctypes structures) to other layers.
2. Delegation:
• Player and GameEngine should call the C functions they logically own, using bindings.lib.
• GameEngine should not perform rendering or contain curses logic.
• GameUI should query the GameEngine for state and call rendering functions only.
3. Clarity and Naming:
• Choose function and method names that reﬂect their intent, not the C API names. (e.g., use
move_player() rather than controller_move_player_within_room().)
• Use Python naming conventions (snake_case for methods and variables, CamelCase for class
names).
4. Modularity:
• Keep ﬁles short and focused.
• Avoid circular imports.
• Model ﬁles should depend only on bindings, not on each other.
5. Error Handling:
• Map C status codes to meaningful Python exceptions using exceptions.py.
• Catch exceptions at the GameEngine or GameUI level where appropriate.
• Never let a ctypes call crash the game loop.
6. Testability:
• Write your methods so they can be tested independently of the UI.
• Separate logic from rendering; for example, move_player() should not print anything or draw
the screen.
7. C Integration
• The C game_engine remains the only interface between Python and C.
• Your Python model must not replicate C-side data (rooms, player, pushables).
• If you need new data from the game_engine, create and expose a new C function and call it
from Python.

Locked Doors (Switches)
When switches are conﬁgured in the world generation settings, datagen creates gated portals that cannot
be traversed until unlocked. A gated portal in a room is linked to a pressure plate switch in that same room.
A switch is activated when a pushable object is placed (pushed) on top of it. The portal remains locked
until its linked switch has a pushable on it. If the pushable is moved oﬀ the switch, the portal should lock
again. How you visually distinguish a locked portal from an open one is up to you.
• Portal and Switch structures already exist in the .h ﬁles and in the charset
• You may need to refactor your movement code to simplify the addition of switches.

Room-Speciﬁc Reset/Undo
Implement two additional undo mechanisms. Players can undo their most recent move, returning to the
previous position and restoring any treasures that were just collected or pushables that were just moved.
Alternatively, players can reset the current room to its initial state returning all pushables and treasures to
their original positions within that room while preserving the state of all other rooms and the player’s overall
treasure count from other rooms. This allows players to recover from mistakes without restarting the entire
game.
• You may need to create c-side functionality to reset a room to original state.
• You will need to be able to undo all types of moves, including those involving treasure and pushables.
